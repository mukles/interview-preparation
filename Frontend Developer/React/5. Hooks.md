

# Question 4: What are Hooks? Why were Hooks introduced?

---

## Mid-Level Answer

> Hooks are functions introduced in **React 16.8** that allow functional components to use React features such as state, lifecycle behavior, context, and refs without writing class components.
> 
> Before Hooks, state management and lifecycle methods like `componentDidMount` and `componentDidUpdate` were only available in class components. Hooks made functional components more powerful and simplified code reuse and maintenance.
> 
> Hooks also make it easier to extract reusable logic into **custom hooks**, which helps keep components clean and avoids duplicating business logic.

---

## Common Built-in Hooks

You should mention the most common ones:

- `useState` – Manage local state.
    
- `useEffect` – Handle side effects.
    
- `useContext` – Access context values.
    
- `useMemo` – Memoize expensive calculations.
    
- `useCallback` – Memoize function references.
    
- `useRef` – Store mutable values or access DOM elements.
    
- `useReducer` – Manage complex state logic.
    

---

## Example

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      {count}
    </button>
  );
}
```

`useState` lets a functional component manage state without using a class component.

---

## Real Example (Your Experience)

> "In the Brritto AI Tutor project, I used `useState` for local UI state, `useEffect` for fetching data and synchronizing side effects, `useRef` for scrolling the chat to the latest message, and `useMemo` to avoid recalculating expensive derived values. We also created custom hooks to encapsulate reusable logic, which made components easier to maintain."

# Follow-up Question 1

## What are the Rules of Hooks?

This is very common.

### Answer

There are two main rules:

### 1. Only call Hooks at the top level.

❌ Wrong

```jsx
if (isLoggedIn) {
  useEffect(() => {});
}
```

✅ Correct

```jsx
useEffect(() => {
  if (isLoggedIn) {
    // ...
  }
}, [isLoggedIn]);
```

---

### 2. Only call Hooks from:

- React function components
    
- Custom Hooks
    

Not from:

- regular JavaScript functions,
    
- loops,
    
- nested functions,
    
- or class components.
    

---

# Follow-up Question 2

## Why can't we call Hooks inside loops or conditions?

Expected answer:

> React identifies each Hook by the order in which it's called during rendering. If a Hook is called conditionally or inside a loop, that order can change between renders. React would then associate state with the wrong Hook, leading to bugs. Calling Hooks consistently at the top level ensures React can correctly preserve state across renders.

---

# Follow-up Question 3

## How does React know which `useState` belongs to which variable?

This is a favorite mid-level question.

Expected answer:

> React doesn't identify Hooks by variable names. It relies on the order in which Hooks are called. As long as the order stays the same on every render, React can correctly associate each Hook with its stored state.

# 1. What is `useState`?

### Mid-Level Answer

> `useState` is a React Hook that allows functional components to manage local state. It returns an array with two values: the current state and a setter function to update that state.
> 
> When the setter function is called, React schedules a state update. React then re-renders the component and updates the UI if necessary.

Example:

```jsx
const [count, setCount] = useState(0);
```

- `count` → current state
    
- `setCount` → updates the state
    

---

# 2. What happens internally when you call `setState`?

This is one of the most common mid-level questions.

Example:

```jsx
const [count, setCount] = useState(0);

setCount(count + 1);
```

### Answer

> When `setCount()` is called, React **does not immediately update the state**. Instead, it schedules a state update. React adds this update to its internal update queue and decides when to process it. During the next render, React calculates the new state, re-renders the component, creates a new Virtual DOM, compares it with the previous Virtual DOM through reconciliation, and updates only the necessary parts of the Real DOM.

# 3. Are state updates synchronous or asynchronous?

### Interview Answer

> State updates are **scheduled**, not applied immediately. Because of that, you shouldn't expect the state variable to have its new value immediately after calling the setter. React batches updates for better performance and applies them during the next render.

Example:

```jsx
const [count, setCount] = useState(0);

function handleClick() {
  setCount(count + 1);

  console.log(count);
}
```

Output:

```
0
```

not

```
1
```

because the update hasn't been processed yet.

---

# 4. What is Batching?

### Interview Answer

> Batching is React's optimization where multiple state updates are grouped into a single render. Instead of re-rendering after every individual update, React processes them together, reducing unnecessary work and improving performance.

Example:

```jsx
setName("Mukles");
setAge(24);
setCity("Dhaka");
```

Without batching:

```
Render
Render
Render
```

With batching:

```
Render
```

only once.

### React 18

React 18 introduced **automatic batching**, which applies to many asynchronous contexts as well, not just React event handlers.

---

# 5. What is Re-rendering?

### Interview Answer

> A re-render is the process where React executes a component function again to produce an updated Virtual DOM. Re-rendering does not necessarily mean the Real DOM will be updated. React first compares the new Virtual DOM with the previous one and updates the Real DOM only if there are actual differences.

Triggers for a re-render include:

- State changes
    
- Props changes
    
- Context changes
    
- Parent component re-rendering (unless optimizations like `React.memo` prevent unnecessary child renders)
    

---

# 6. What is Reconciliation?

### Interview Answer

> Reconciliation is React's process of comparing the previous Virtual DOM with the new Virtual DOM after a render. React uses a diffing algorithm to identify the minimal set of changes and applies only those updates to the Real DOM.

Example:

Old:

```jsx
<h1>0</h1>
```

New:

```jsx
<h1>1</h1>
```

React updates only the text node instead of recreating the entire `<h1>` element.

---

# 7. What is the Virtual DOM?

### Interview Answer

> The Virtual DOM is a lightweight JavaScript representation of the Real DOM. React uses it to efficiently determine what has changed between renders. When state or props change, React creates a new Virtual DOM, compares it with the previous one, and updates only the necessary parts of the Real DOM.

# Complete Interview Flow

An interviewer may ask:

> **What happens when you click this button?**

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      {count}
    </button>
  );
}
```

A strong answer would be:

> "When the button is clicked, `setCount(count + 1)` schedules a state update. React doesn't immediately modify the state. Instead, it queues the update and processes it during the next render. React executes the component function again with the new state, creating a new Virtual DOM. It then performs reconciliation by comparing the new Virtual DOM with the previous one. Since only the text inside the button has changed, React updates just that text in the Real DOM. The browser then paints the updated UI."

---

# ⭐ Interview Tip

One detail that often impresses interviewers is avoiding the statement:

> ❌ "React updates the DOM."

A more precise explanation is:

> ✅ "React re-renders the component to produce a new Virtual DOM. Through reconciliation, it determines the minimal changes required and then updates only the affected parts of the Real DOM."

That wording demonstrates a clear understanding of React's rendering process and is what many mid-level interviewers are looking for.
