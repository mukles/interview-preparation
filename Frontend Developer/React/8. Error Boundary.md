# Interview Question

## What is an Error Boundary?

### Mid-Level Answer

> An **Error Boundary** is a React component that catches JavaScript errors that occur during rendering, in lifecycle methods, and in constructors of its child component tree. Instead of allowing the entire React application to crash, it displays a fallback UI and optionally logs the error for monitoring and debugging.
> 
> Error Boundaries improve the user experience by isolating failures to a specific part of the application. For example, if one widget on a dashboard crashes, the rest of the application can continue working.

---

# Why do we need Error Boundaries?

Without an Error Boundary:

```text
Dashboard
├── User Profile
├── Statistics
├── Notifications ❌ throws error
└── Chat
```

If `Notifications` throws during rendering:

❌ The whole React tree may unmount.

---

With an Error Boundary:

```text
Dashboard
├── User Profile
├── Statistics
├── Error Boundary
│      └── "Something went wrong."
└── Chat
```

Only the wrapped section shows the fallback UI.

---

# Example

```jsx
class ErrorBoundary extends React.Component {
  state = { hasError: false };

  static getDerivedStateFromError(error) {
    return { hasError: true };
  }

  componentDidCatch(error, errorInfo) {
    console.error(error, errorInfo);
    // Send error to Sentry, LogRocket, etc.
  }

  render() {
    if (this.state.hasError) {
      return <h2>Something went wrong.</h2>;
    }

    return this.props.children;
  }
}
```

Usage:

```jsx
<ErrorBoundary>
  <UserDashboard />
</ErrorBoundary>
```

# Lifecycle Methods Used

React provides two lifecycle methods for Error Boundaries.

### `static getDerivedStateFromError()`

Purpose:

- Update state
    
- Show fallback UI
    

Example:

```jsx
static getDerivedStateFromError(error) {
    return {
        hasError: true
    };
}
```

---

### `componentDidCatch()`

Purpose:

- Logging
    
- Reporting
    

Example:

```jsx
componentDidCatch(error, info) {
    logError(error, info);
}
```

# What can Error Boundaries catch?

✅ Rendering errors

```jsx
throw new Error("Oops");
```

during render.

---

✅ Errors in lifecycle methods.

---

✅ Errors in constructors.

---

# What can they NOT catch?

This is asked very frequently.

They **cannot** catch:

### Event handlers

```jsx
<button onClick={() => {
    throw new Error();
}}>
```

Use:

```jsx
try {
} catch(e) {
}
```

inside the event handler if needed.

---

### Async code

```jsx
setTimeout(() => {
    throw Error();
});
```

Not caught.

---

### Promise rejections

```jsx
fetch(...)
```

Not caught automatically.

Handle them with:

```jsx
try {
    await fetch();
} catch (error) {
}
```

or `.catch()`.

---

### Server-side rendering

Traditional client-side Error Boundaries don't catch errors during server-side rendering.

# Follow-up Questions

### 1. Can functional components be Error Boundaries?

Expected answer:

> Not directly. Error Boundaries must currently be implemented as class components. However, functional components can use third-party libraries, such as `react-error-boundary`, which provide a Hook-friendly API while using an Error Boundary internally.

---

### 2. Why did React introduce Error Boundaries?

Expected answer:

> Before React 16, an uncaught rendering error could leave the UI in an inconsistent state. React 16 introduced Error Boundaries so applications could fail gracefully by showing fallback UI instead of crashing the entire interface.

---

### 3. Where would you place Error Boundaries?

Good answer:

> I typically place them around independent sections of the UI, such as routes, dashboards, widgets, or third-party components, rather than wrapping the entire application in a single Error Boundary. This isolates failures and keeps unaffected parts of the app usable.

---

### 4. Why not wrap the entire app with one Error Boundary?

A single top-level Error Boundary is useful as a safety net, but if it catches an error, the whole application may show the same fallback UI. Placing additional boundaries around isolated features provides a better user experience because one failing component doesn't take down unrelated parts of the interface.

---

# Interview Tip

A strong mid-level answer doesn't stop at _"it catches errors."_ Mention:

- **what it catches** (rendering, lifecycle methods, constructors),
    
- **what it doesn't catch** (event handlers, async code, Promise rejections, SSR),
    
- **where you'd use it** (feature-level isolation),
    
- and **how you'd log errors** (for example, with Sentry or another monitoring service).
    

That level of detail shows both conceptual understanding and production experience.
